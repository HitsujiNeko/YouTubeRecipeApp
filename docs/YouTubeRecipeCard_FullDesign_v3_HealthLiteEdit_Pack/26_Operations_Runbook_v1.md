# 26 運用Runbook v1

作成日: 2026-03-05

## 26.1 目的
- 障害時の初動と復旧を標準化し、サービス影響を最小化する。

## 26.2 優先インシデント
- P1: 全面停止、データ破損、認可不備
- P2: import失敗率上昇、429/503増加
- P3: 一部機能劣化

## 26.3 429対応手順
1. 監視アラート確認（provider, route, actor type）。
2. 一時的にレート制限を強化。
3. UIでリトライ待機メッセージを表示。
4. バックオフ値を引き上げ（最大待機時間2倍）。
5. 復旧後に制限を段階的に戻す。

## 26.4 503対応手順
1. 外部依存障害を判定（YouTube/OpenAI/Supabase）。
2. サーキットブレーカーをON。
3. fallback応答（best-effort + retryable=true）を返す。
4. 依存先復旧確認後、ブレーカーを段階解除。

## 26.5 鍵ローテーション手順
- 対象: `OPENAI_API_KEY`, `YOUTUBE_DATA_API_KEY`, `SUPABASE_SERVICE_ROLE_KEY`
1. 新キー発行
2. Secret Store更新（Vercel/Supabase）
3. ステージング疎通確認
4. 本番反映
5. 旧キー無効化
6. 監査ログ記録

## 26.6 監視・通知
- 通知先: Slack #ops-alert
- アラート閾値:
  - 5xx率 > 3%（5分）
  - 429率 > 5%（10分）
  - p95 > 3s（10分）

## 26.7 インシデント後対応
- 24時間以内に暫定報告。
- 72時間以内にポストモーテム。
- 恒久対策をIssue化し、優先度を設定。
